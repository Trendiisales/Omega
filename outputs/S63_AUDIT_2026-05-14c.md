## Why the XauTrendFollow trio is the only viable candidate

The three `XauTrendFollow*Engine` siblings share the same
management shape (confirmed by reading the `_manage_open()` body
in each):

```cpp
// XauTrendFollow{2h,4h,D1}Engine.hpp _manage_open():
//   - Update MFE/MAE on every tick (mid-price favourable distance).
//   - If long: exit at bid when bid <= sl_px (SL_HIT) or bid >= tp_px (TP_HIT)
//   - If short: exit at ask when ask >= sl_px (SL_HIT) or ask <= tp_px (TP_HIT)
//   - No trail, no BE-arm, no cold-loss-cut, no timeout.
```

This is the same failure profile that motivated S63 on the
VWAPReversionEngine class — a fixed bracket with no in-flight
protection means a trade that goes adverse at entry will ride to
the SL_atr distance even if the adverse move is unambiguous within
seconds. The cold-loss cut would cap that scenario.

**But — these are trend-follow engines.** A break-even ratchet
(BE_ARM_PCT) on a trend-follow profile carries real winner-
amputation risk: trades that arm BE and then pull back to the
ratcheted SL would have been net winners had they been left to
ride to TP. The VWR profile (mean-reversion + fixed timeout) doesn't
suffer this because the trade is meant to revert quickly; the
trend-follow profile is the opposite.

The empirically-correct path is the same one part-K/L took for the
VWR US500 / NQ / EURUSD instances:
1. Sweep `LOSS_CUT_PCT` alone (no BE ratchet) on the existing tape
   for each XauTrendFollow timeframe (2h / 4h / D1). Confirm or
   reject net edge.
2. Separately sweep `BE_ARM_PCT` + `BE_BUFFER_PCT` on top of (1).
   Likely to be net-negative on a trend-follow profile, but the
   data should drive the decision.
3. Wire whichever subset of S63 fields passed step 1+2 into the
   class.

Until those sweeps run, **State E is the correct state for the
XauTrendFollow trio.** Adding S63 fields without empirical
backing would be a config change masquerading as protection.

## Why the other 5 engines should not get S63

### UstecTrendFollowHtfEngine

Existing protection (engine_init.hpp:1147-1156):

```cpp
g_ustec_tf_htf.be_trigger_atr   = 1.0;    // arm BE at +1.0 * ATR MFE
g_ustec_tf_htf.be_cost_buffer_pts = 0.50; // BE level = entry + 0.50 pts
g_ustec_tf_htf.trail_after_be   = true;   // trail SL after BE arms
g_ustec_tf_htf.trail_atr_mult   = 0.75;
g_ustec_tf_htf.min_atr_floor    = 5.0;
g_ustec_tf_htf.max_atr_ceil     = 0.0;    // disabled
g_ustec_tf_htf.daily_loss_limit = 0.0;    // disabled
g_ustec_tf_htf.max_consec_losses = 0;     // disabled
g_ustec_tf_htf.max_bars_held    = 0;      // disabled
```

`be_trigger_atr` is mechanically equivalent to S63's `BE_ARM_PCT`
(arm break-even at threshold MFE) with a different unit (ATR
multiple vs entry-relative pct). Adding `BE_ARM_PCT` on top would
result in two BE-arming code paths competing for the same trade.

The part-M handoff predicted "may inherit the same S63 pattern
when it lands" — the correct read is that the HTF sibling already
had its own BE mechanism baked in, which is why part-L's S63 wiring
on the 5m sibling did NOT replicate to the HTF.

A cold-loss-cut (`LOSS_CUT_PCT` only, no BE) could in principle add
value here, since the BE-arm doesn't fire until +1.0 ATR MFE and a
trade that immediately goes adverse has no protection from the
existing mechanism. But this would be a one-field S63 add, not the
full pair, and should be validated by the same sweep workflow the
XauTrendFollow trio needs.

### MacroCrashEngine

The engine identity is the STEP1/STEP2 banking ladder
(MacroCrashEngine.hpp:81-82, 774-806):

- STEP1 triggers at +$200 open_pnl: banks a fraction of size + moves
  SL to break-even.
- STEP2 triggers at +$400 open_pnl: banks another slice.

This is a different protection model from S63 — dollar-threshold
rather than pct-of-trade — designed around the crash-event trade
profile (low frequency, large MFE potential, drawdown tolerated to
catch the move). Adding S63's `LOSS_CUT_PCT` would pre-empt the
crash from developing. The engine was explicitly retuned in S42
(2026-04-02) to widen the dollar thresholds; S63 would walk that
back.

### H1SwingEngine + H4RegimeEngine

Both engines use the gold-swing parameter pack
(`make_h1_gold_params` / `make_h4_gold_params`) with:

- `tp1_mult` — partial TP1 at +tp1_mult * H1_ATR (banks 50% of size).
- `tp2_trail_arm_mult` — trail arms at this MFE.
- `tp2_trail_dist_mult` — trail-SL distance for remaining 50%.

The trail-SL is the BE-equivalent: once MFE reaches
`tp2_trail_arm_mult * ATR`, the SL ratchets up behind price by
`tp2_trail_dist_mult * ATR`. Adding S63 BE_ARM on top would compete
with the trail-arm mechanism in the same way as the UstecHtf case.

H1Swing is currently `enabled = false` (engine_init.hpp:756) so S63
wiring would be a no-op anyway. H4Regime is enabled and live in
shadow.

### EMACrossEngine + RSIExtremeTurnEngine

Both are disabled engines (RSIExtreme `enabled = false` per
engine_init.hpp:42; EMACross shadow-only per `kShadowDefault`).
Both were cited in the part-M memo as having returned 0/153 combos
profitable in their respective sweeps. Adding S63 to either is moot
— the engines don't fire.

   (N/A).

established. The full remaining ~13 engines should fit in a single
