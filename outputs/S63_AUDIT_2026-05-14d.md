## Why none of these are wiring candidates

The seven SymbolEngines / five FX BreakoutEngines / five cross-asset
engines all share the same protection profile in different dialects:

```
[entry] -> [trail-arm at X% of TP MFE] -> [trail-SL at Y% of TP/SL distance behind peak]
       +-> [BE arm at threshold]
       +-> [hard MAX_HOLD_SEC timeout]
       +-> [regime/event-specific flatten override]
```

This is **richer than** S63's `LOSS_CUT_PCT` + `BE_ARM_PCT` pair.
Specifically:
- BE_ARM is subsumed by their explicit BE-arm code.
- The cold-loss-cut intent (catch immediate-adverse trades) is
  partially handled by the engine-specific timeouts (300-1800s)
  on the cross-asset engines plus the REGIME_FLIP flatten path on
  the SymbolEngines / FX BreakoutEngines.

Adding S63 fields on top of these would create two BE-arming code
paths competing for the same trade — exactly the failure mode the
part-N memo flagged for `UstecTrendFollowHtfEngine`. The decision
to wire S63 was always meant to be a one-size-fits-engine-shape
call, and this shape is already covered.

The `TrendPullbackEngine` is the strongest case for "richer than
S63" — it has BE-arm AT 1× ATR MFE, trail-arm at 2× ATR, ATR-floored
trail distance, AND a partial exit at 0.95×TP. Wiring S63 LOSS_CUT
on top would prematurely cut trades that the ATR trail is designed
to ride.

The two scalpers are special cases:
- `GoldMidScalperEngine` has its own MFE-based trail tuned at S53
  (2026-05-01) — adding S63 LOSS_CUT would cut sub-second trades
  before the trail had a chance to engage.
- `GoldMicroScalperEngine` is an inert stub — no S63 needed because
  no trades fire.

## Methodology note (carried forward to wrapper audit)

Same idiom as parts L → M → N. For each engine class:

1. Grep header for `LOSS_CUT_PCT|BE_ARM_PCT|BE_BUFFER_PCT`.
   - Zero hits → State E. Proceed to step 2.
   - Any hits → State A/B/C/D based on management-path + init override.
2. For State E: identify existing in-flight protection by reading
   the `manage` / `_manage_open` / `on_tick`-management function
   body. Look for:
   - Trail (peak-MFE-relative or ATR-relative).
   - BE arm at threshold.
   - Timeout / MAX_HOLD.
   - Engine-specific overrides (regime-flip, event-flatten, partial,
     pyramid-add-on, TP extension).
3. S63 fit verdict:
   - **Candidate** — only TP_HIT/SL_HIT/FORCE_CLOSE, no trail/BE/timeout.
     Sweep-gated.
   - **NOT a fit** — has overlapping BE/trail mechanism.
   - **Different model** — has incompatible protection (e.g.
     dollar-threshold ladder, partial-TP).
   - **N/A** — engine disabled / culled / inert stub.

The simple-pos audit took 4 sessions (L → O) over the full ~40+
engine inventory. The wrapper audit (BracketEngine, GoldEngineStack,
CandleFlow, portfolios) is the natural next stage and will need
per-wrapper design notes before a state-letter is assignable.
